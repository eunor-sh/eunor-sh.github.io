1.

오늘의 삽질 기록​

최근 Ceph 신규 구축을 진행하면서, 단순히 구성만 맞추는 것보다 네트워크 설계가 성능과 안정성에 훨씬 큰 영향을 준다는 걸 체감했다.
특히 LACP를 “대역폭 합산”이 아니라 “트래픽 분산” 관점에서 다시 볼 필요가 있다고 느껴 정리하게 되었다.

이론​

Ceph에서 Bonding이 필수적인 이유​

Ceph는 구조적으로 네트워크 의존도가 매우 높은 분산 스토리지입니다.
특히 데이터 복제(Replication)와 복구(Recovery) 과정에서 막대한 네트워크 트래픽이 지속적으로 발생합니다.

따라서 단일 NIC 기반 구성은 Ceph와 같은 분산 스토리지 환경에서 네트워크 병목과 장애 리스크를 동시에 유발할 수 있습니다.

이러한 한계를 보완하기 위해 NIC Bonding 구성이 필요하며, 이를 통해 다음과 같은 효과를 얻을 수 있습니다.

  • 고가용성(HA)
    • 다수의 NIC을 하나의 논리 인터페이스로 구성함으로써, 특정 NIC 또는 연결된 스위치에 장애가 발생하더라도 나머지 링크를 통해 트래픽을 지속적으로 처리할 수 있습니다.
  • 대역폭 확장
    • 여러 물리 인터페이스를 병렬로 활용하여 네트워크 처리량을 증가시킬 수 있으며, 이는 OSD 간 데이터 복제 및 백필(backfill) 성능에 직접적인 영향을 미칩니다.

Ceph 클러스터에서 Mode 4(LACP)를 권장하는 이유​

Ceph 환경에서 네트워크는 단순한 데이터 전달 수단이 아니라, 클러스터의 성능과 안정성을 결정하는 핵심 구성 요소입니다.

특히 Ceph는 데이터의 일관성과 내결함성을 보장하기 위해 복제(Replication), 백필(Backfill), 리밸런싱(Rebalancing)과 같은 내부 작업을 지속적으로 수행하며, 이 과정에서 노드 간 대규모 East-West 트래픽이 발생합니다.

이러한 특성으로 인해 네트워크는 다음 두 가지 요구사항을 동시에 만족해야 합니다.

  • 높은 처리량 (High Throughput)
  • 안정적인 링크 유지 및 장애 대응 (High Availability)

Bonding의 여러 모드 중 LACP(Mode 4)는 이 두 가지 요구사항을 동시에 만족할 수 있습니다.

Ceph의 트래픽과 LACP의 궁합​

Ceph는 단일 스트림 기반으로 트래픽이 흐르는 구조가 아니다.
클러스터 내부에서 다수의 OSD가 동시에 통신하면서, 자연스럽게 여러 개의 TCP 세션이 병렬로 생성되는 구조를 가진다.

예를 들어 하나의 Write 요청이 발생하면,

  • 클라이언트 → Primary OSD로 요청 전달
  • Primary OSD → Secondary OSD들로 복제 트래픽 전송
  • Secondary OSD → Primary OSD로 ACK 반환
  • Primary OSD → 클라이언트로 최종 응답

이 과정은 단일 흐름이 아니라, 여러 OSD 간 통신이 동시에 발생하는 구조로 동작한다.
리밸런싱이나 Recovery가 시작되면 이러한 흐름은 클러스터 전체로 확산되며 대규모 트래픽을 유발한다.

클러스터 전체에서 OSD 간 데이터 이동이 동시에 발생하면서, 네트워크는 다수의 세션으로 쪼개진 상태로 동작한다.

결국 Ceph 네트워크의 특징은 명확하다.

특정 노드 간 단일 트래픽이 아니라 여러 노드 간 다수의 연결이 동시에 발생하는 구조

이 구조에서는 네트워크 구성 방식에 따라 성능 차이가 바로 드러난다.

  • 단일 NIC → 모든 트래픽이 하나의 링크로 집중
  • Active-Backup(mode1) → 정상 상황에서도 동일하게 단일 링크 사용
  • Round-robin(mdoe0) → 패킷 단위 분산으로 TCP 성능 저하

반대로 LACP는 flow 단위로 트래픽을 분산시키고 서로 다른 세션을 서로 다른 NIC으로 나누어 처리한다.
동일한 세션은 동일한 경로를 유지하기 떄문에 패킷 순서 또한 보장된다.

이로 인해 Ceph 환경에서는 OSD 간 통신이 여러 NIC으로 자연스럽게 분산되며, 특정 링크에 트래픽이 집중되는 현상이 완화된다.

결과적으로 여러 링크를 동시에 활용하는 구조가 형성되고, 네트워크 자원의 사용 효율 또한 높아진다.

정리하면,

Ceph는 여러 개의 통신 흐름이 동시에 발생하는 구조를 가지며, LACP는 이러한 흐름을 여러 링크로 분산시키는 방식이기 때문에 두 구조는 자연스럽게 맞물린다.

LACP 고려사항​

앞서 살펴본 것처럼 LACP는 Ceph와 같이 다수의 flow가 동시에 발생하는 환경에서 매우 효과적인 방식이다.
하지만 LACP가 모든 상황에서 선형적인 성능 향상을 보장하는 것은 아니다.

LACP는 트래픽을 flow 단위로 분산시키기 때문에, 하나의 flow는 항상 하나의 NIC만 사용하게 된다.

이로 인해 다음과 같은 한계가 발생한다.

  • 단일 대용량 트래픽은 하나의 NIC 대역폭에 제한됨
  • 특정 workload가 소수의 세션에 집중될 경우 분산 효과가 제한됨
  • 기대한 "대역폭 합산" 효과가 나타나지 않을 수 있음

즉, 여러 NIC을 묶었다고 해서 하나의 트래픽이 여러 링크를 동시에 사용하는 구조는 아니다.

LACP는 대역폭을 단순히 합치는 기술이 아니라, 트래픽을 분산시키는 기술이다.

다만 Ceph 환경에서는 이러한 한계가 크게 문제되지 않는다.

Ceph는 구조적으로 다수의 OSD 간 통신이 동시에 발생하며, 자연스럽게 여러 개의 flow가 생성되는 특성을 가진다.
이로 인해 각 flow가 서로 다른 NIC으로 분산되면서 전체 네트워크 대역폭을 효과적으로 활용할 수 있다.

궁금증 1. Client에서 Ceph으로 Write한다는건 단일 세션이고 이는 단일 flow이기 때문에 1개의 NIC만 쓸텐데 그럼에도 이점이 있는가?
분석

단일 클라이언트가 Ceph로 Write를 수행하는 경우, 해당 트래픽은 하나의 TCP 세션으로 구성되며 하나의 flow로 동작한다.
이 경우 LACP는 해당 트래픽을 여러 NIC으로 분산시키지 않으며, 실제로 하나의 NIC만 사용하게 된다.

따라서 단일 Write 트래픽만을 기준으로 보면 LACP의 대역폭 확장 효과는 제한적으로 나타난다.

하지만 Ceph는 단순히 클라이언트 요청만 처리하는 구조가 아니다.

클라이언트 Write 이후, Primary OSD는 데이터를 다수의 Secondary OSD로 복제하게 되며 이 과정에서 여러 개의 통신이 동시에 발생한다.
또한 리밸런싱이나 Recovery 상황에서는 OSD 간 대규모 데이터 이동이 클러스터 전반에서 동시에 이루어진다.

이러한 내부 트래픽은 다수의 flow로 구성되며, LACP를 통해 여러 NIC으로 자연스럽게 분산된다.

결과적으로 LACP의 효과는 단일 클라이언트 트래픽보다는 Ceph 내부 통신에서 크게 나타나며,
특히 복제 및 리밸런싱 상황에서 네트워크 병목을 완화하는 데 중요한 역할을 한다.

정리하면,단일 클라이언트 Write 관점에서는 LACP의 대역폭 확장 효과가 제한적으로 나타나지만,
Ceph의 특성상 복제 및 리밸런싱과 같은 내부 통신에서 다수의 flow가 동시에 발생하기 때문에 LACP의 분산 효과는 충분히 활용된다.

따라서 LACP는 단일 트래픽의 성능 향상을 위한 기술이라기보다,
Ceph 전체 네트워크의 처리 효율과 안정성을 확보하기 위한 필수적인 구성 요소로 보는 것이 적절하다.

LACP Load Balancing 정책 (Hash Policy)​

LACP 분산 효율은 Hash 정책에 따라 크게 달라진다.
동일한 LACP 구성이라도 어떤 기준으로 flow를 분산시키느냐에 따라 실제 체감 성능은 크게 차이난다.

일반적으로 사용되는 Hash 정책은 다음과 같다.

Hash 기반 NIC 선택 방식

LACP의 Load Balancing은 트래픽을 무작위로 분산하는 방식이 아니라,
패킷 헤더의 일부 필드를 기반으로 Hash 값을 계산한 뒤 해당 값을 특정 NIC에 매핑하는 방식으로 동작한다.

Hash 계산에 사용되는 필드는 설정된 정책(xmit_hash_policy)에 따라 달라진다.

  • layer2: MAC 주소 기반
  • layer3+4: IP 주소 + TCP/UDP Port 기반

간단히 표현하면 다음과 같다.

NIC_index = hash(선택된 필드 값들) % NIC 개수

이때 동일한 입력값에 대해서는 항상 동일한 Hash 값이 생성된다.

즉, 동일한 조건의 트래픽은 항상 동일한 NIC으로 전달되며 입력값이 다른 트래픽만 다른 NIC으로 분산된다.

이 동작의 핵심은 다음과 같다.

  • 분산 기준은 패킷 단위가 아니라 헤더 필드 조합이다
  • 동일한 조건의 트래픽은 항상 동일한 경로를 유지한다
  • 서로 다른 조건의 트래픽만 NIC 간 분산된다

정리하면,

LACP의 Hash는 특정 트래픽을 특정 NIC으로 매핑하는 결정 규칙이며, 동일한 조건에서는 항상 동일한 결과를 반환한다.

  • layer2: MAC 주소 기반
  • layer3+4: IP + Port 기반

Ceph 환경에서는 layer3+4 기반 Hash 정책을 사용하는 것이 사실상 필수에 가깝다.

그 이유는 다음과 같다.

  • OSD 간 통신 시 동일한 MAC 주소를 공유하는 경우가 많음
  • layer2 기반 Hash는 특정 NIC으로 트래픽이 쏠릴 가능성이 높음
  • layer3+4는 TCP Port까지 포함하여 더 세밀한 분산 가능

이로 인해 NIC 간 트래픽 분산 균동도가 향상되고 특정 링크에 부하가 집중되는 현상을 줄일 수 있다.

실습 목표 및 검증 기준​

실습의 목표는 2개의 NIC을 LACP Bonding(802.3ad)으로 구성한 이후,
네트워크의 고가용성(HA)과 트래픽 분산 특성을 실제 환경에서 검증하는 것이다.

검증 목표는 다음과 같다.

  1. Flow 기반 분산 동작 확인
  • 단일 TCP 세션에서는 하나의 NIC만 사용되는지 확인한다.
  • 다중 TCP 세션에서는 여러 NIC으로 트래픽이 분산되는지 확인한다.
  1. 성능 검증
  • 단일 세션과 다중 세션 환경에서의 네트워크 처리량 차이를 비교한다.
  • 특히 다중 flow 환경에서 전체 대역폭이 확장되는지 확인한다.

결과적으로, 본 실습의 목표는 LACP 구성이 Ceph 환경에서 단순한 대역폭 확장 이상의 의미를 가지는지,
즉 다수의 트래픽이 발생하는 상황에서 실제 성능 향상과 장애 대응 측면에서 유효한 설계인지 검증하는 데 있다.

검증 및 테스트​

Flow 기반 분산 동작 확인과 성능 검증​

25G NIC 2개를 LACP Bonding으로 구성한 만큼 기존 단일 25G 환경 대비 실제 네트워크 처리량(throughput)이 증가하는지 확인해보자.

이론적으로는 두 링크가 집계되면서 최대 50Gbps 수준까지 확장이 가능하며 테스트 환경에서는 40Gbps 이상의 처리량이 기대된다.

단, LACP는 트래픽을 패킷 단위가 아닌 flow(세션) 단위로 분산하는 구조이기 때문에, 단일 TCP 연결에서는 하나의 NIC만 사용하게 된다.

따라서 단일 flow 환경에서는 최대 25Gbps 수준의 성능만 나오는지도 확인해볼 포인트이다.

이를 검증하기 위해 iperf3를 활용하여 단일 flow와 다중 flow 환경을 각각 비교 테스트한다.

tip

네트워크 성능은 항상 송신(TX)과 수신(RX) 경로 중 더 낮은 대역폭을 가진 구간이 병목이 된다.

예를 들어, LACP로 25G x2 구성을 하더라도 상대 측이 단일 25G NIC이라면 전체 처리량은 25Gbps를 초과할 수 없다.

따라서 LACP 성능을 제대로 확인하기 위해 충분한 대여곡을 가진 환경(예: 50G NIC 이상)을 기준으로 100G → 25G × 2(LACP) 방향으로 테스트하는 것이 적절하다.

반대로 25G 단일 NIC 환경에서 테스트를 수행할 경우, LACP 구성 여부와 관계없이 25Gbps 이상의 성능을 확인할 수 없다.

마치며​

구성해보면서 느낀 점은, LACP를 무조건 먼저 고려하기보다는 현재 가지고 있는 자원에 맞춰 설계를 가져가는 게 더 중요하다고 느껴졌다.

지금처럼 듀얼 NIC 1개와 케이블 2개뿐인 상황이라 이를 LACP로 묶기보다는 서비스망과 스토리지망으로 분리해서 사용하는 구조가 Ceph에서는 더 적절하게 느껴졌다.
특히 내부 트래픽이 많은 Ceph 환경에서는 망을 분리하는 쪽이 더 안정적인 구성이라 생각이 된다.

반대로 NIC과 케이블 여유가 충분하다면, LACP로 묶어서 서비스망 스토리지망을 구성하는게 더 좋은 선택지라고 생각한다.

나중에 실제 서비스에서 트래픽이 증가하거나 네트워크 병목이 발생하는 상황이 온다면, 그 때 LACP 구성을 추가하면서 케이블을 확장하는 방향으로 가져가는 것도 충분히 고려해볼 수 있을 것 같다.
