Troubleshooting)스토리지 장애로 인한 Terminating Pod 수동 삭제

April 20, 2026 · 읽는데 약 6분
Eunor-sh
Infrastructure Researcher

⚡️ 요약 (TL;DR)​

백엔드 스토리지 롤백 등으로 인해 마운트 정보가 깨져 Terminating 상태에 갇히 Pod를, Shim 프로세스 사살 및 containerd 런타임 제어를 통해 물리적으로 제거하여 해결했습니다.

~ ➜
BASH
# 1. 장애 파드 위치 및 상세 정보 확인
# 여기서 해당 파드가 어느 NODE에 배정되었는지 확인합니다.
~ ➜ kubectl get po [POD_NAME] -o wide 

# 2. 해당 노드 접속 후 컨테이너 식별 및 API 객체 강제 삭제
node19 ~ ➜ docker ps | grep [POD_NAME]
# API 객체부터 삭제해야 Kubelet이 더 이상 해당 파드를 복구하려 하지 않습니다.
~ ➜ kubectl delete pods [POD_NAME] --force

# 3. 고립된 컨테이너의 관리자(Shim) 프로세스 PID 확인 
# docker ps로 확인한 [CONTAINER_ID]를 사용하여 shim 프로세스를 찾습니다.
node19 ~ ➜ ps aux | grep [CONTAINER_ID]

# 4. Shim 프로세스 강제 종료
# 런타임이 물고 있는 연결 고리를 끊어 docker rm이 가능하도록 만듭니다.
node19 ~ ➜ kill -9 [SHIM_PID]

# 5. docker 명령어로 최종 삭제 확인
node19 ~ ➜ docker rm -f [CONTAINER_ID]

🚨 문제 상황 (Issue)​

Kubernetes 클러스터 운영 중 특정 Pod가 삭제되지 않고 0/1 Terminating 상태로 유지되는 현상이 발생했습니다.

🔍 원인 파악 (Insight)​

  • 스토리지 실 데이터 부재: 외부 요인으로 백엔드 스토리지 데이터가 삭제된 후 특정 시점으로 롤백되면서, Pod가 마운트하고 있던 실제 데이터 블록이나 엔드포인트가 사라졌습니다.

  • 언마운트 논리적 고립: Kubelet은 Pod 삭제 시 볼륨을 먼저 언마운트해야 하지만, 스토리지 엔드포인트에 실제 데이터가 없으니 마운트 해제 절차(Cleanup)를 정상적으로 완료하지 못하고 Terminating에 빠짐.

  • 자원 회수 불능: kubectl delete --force로 API 객체는 지울 수 있으나, 노드 레벨에서는 Kubelet이 볼륨 정리에 실패했으므로 컨테이너 런타임에 종료 신호를 보내지 못해 실제 자원(CPU, Memory, Mount Point) 점유가 유지

~ ➜ kubectl get pods

pod-696d70616374-rlnxto-668b9cb996-j64q5                     0/1     Terminating   0              35h
pod-696d70616374-rlnxto-6f94746d57-cv4c7                     0/1     Terminating   1              2d5h

🛠️ 해결 방법 (Fix)​

1개의 Pod(=pod-696d70616374-rlnxto-668b9cb996-j64q5)를 수동으로 삭제하는 것을 다룹니다.

1. 컨테이너 식별

먼저 문제가 발생한 노드와 컨테이너 ID를 확인합니다.

  • Terminating상태의 POD가 실제 배치된 노드와 컨테이너 식별
~ ➜ kubectl get pods pod-696d70616374-rlnxto-668b9cb996-j64q5 -o wide

NAME                                           READY   STATUS        RESTARTS   AGE   IP       NODE             NOMINATED NODE   READINESS GATES
pod-696d70616374-rlnxto-668b9cb996-j64q5       0/1     Terminating   0          36h   <none>   node19           <none>           <none>
  • NODE=node19를 식별했으므로 해당 노드에 접속 후 Pod명의 docker 컨테이너를 조회합니다.
node19 ~ ➜ docker ps | grep pod-696d70616374-rlnxto-668b9cb996-j64q5

fdb38c96b7ec   k8s.gcr.io/pause:3.5   "/pause"                 37 hours ago   Up 37 hours             k8s_POD_pod-696d70616374-rlnxto-668b9cb996-j64q5_pod_77f17d3d-8f49-4cd3-9768-1ade08f2724d_0

2. Kubernetes POD 강제 삭제 쿠버네티스 컨트롤 플레인에서 해당 객체를 먼저 밀어 냅니다.

note

해당 상태Terminating에서는 kubectl delete 명령어로는 삭제가 되지 않습니다.
pod deleted에서 무한 루프가 걸립니다.

~ ➜ k delete po pod-696d70616374-rlnxto-668b9cb996-j64q5

pod "pod-696d70616374-rlnxto-668b9cb996-j64q5" deleted

--force 옵션을 추가하여 POD를 삭제합니다.
단, Kubernetes API를 통해 삭제가 된 것이지 실제 POD 컨테이너가 삭제된 것은 아닙니다.

~ ➜ k delete po pod-696d70616374-rlnxto-668b9cb996-j64q5 --force

warning: Immediate deletion does not wait for confirmation that the running resource has been terminated. The resource may continue to run on the cluster indefinitely.
pod "pod-696d70616374-rlnxto-668b9cb996-j64q5" force deleted

node19번에서 Pod 컨테이너를 조회해보면 삭제되지 않고 남아있는 것을 확인할 수 있습니다.
이 경우 실제 자원을 점유하고있기때문에 수동으로 삭제처리를 진행해주어야합니다.

node19 ~ ➜ docker ps | grep pod-696d70616374-rlnxto-668b9cb996-j64q5

fdb38c96b7ec   k8s.gcr.io/pause:3.5   "/pause"                 37 hours ago   Up 37 hours             k8s_POD_pod-696d70616374-rlnxto-668b9cb996-j64q5_pod_77f17d3d-8f49-4cd3-9768-1ade08f2724d_0

3. 컨테이너 및 프로세스 정보 추출 노드에 접속하여 Docker가 인지하고 있는 컨테이너와 실제 프로세스 ID(PID)를 환경 변수로 선업합니다.

node19 ~ ➜ CONTAINER=fdb38c96b7ec
node19 ~ ➜ PID=$(sudo docker inspect --format '{{.State.Pid}}' $CONTAINER)
node19 ~ ➜ echo $CONTAINER

fdb38c96b7ec

node19 ~ ➜ echo $PID

620780

4. 단계별 컨테이너 삭제 시도 (실패 케이스 포함)

Step 4-1. docker kill, rm을 통한 컨테이너 삭제 (실패)

node19 ~ ➜ docker kill $CONTAINER

Error response from daemon: Cannot kill container: fdb38c96b7ec: tried to kill container, but did not receive an exit event

node19 ~ ➜ docker rm -f $CONTAINER

Error response from daemon: Could not kill running container 
fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f, cannot remove - tried to kill container, but did not receive an exit event

Step 4-2. 컨테이너 프로세스를 추적하여 SIGKILL (실패)

컨테이너 PID(Process ID) 조회

node19 ~ ➜ ps $PID

    PID TTY      STAT   TIME COMMAND
 620780 ?        Ss     0:00 [pause]

프로세스 강제 종료 (SIGKILL) 후 재조회

  • SIGKILL을 통해서는 삭제되지 않는 것을 확인
node19 ~ ➜ kill -9 $PID

node19 ~ ➜ ps $PID

    PID TTY      STAT   TIME COMMAND
 620780 ?        Ss     0:00 [pause]

4-3. 'ctr'을 통해 삭제 (실패)

컨테이너 조회

node19 ~ ➜ ctr -n moby task ls | grep $CONTAINER

fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f    620780     RUNNING

컨테이너 삭제

node19 ~ ➜ ctr -n moby task rm -f fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f

ERRO[0000] unable to delete fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f  error="unknown error after kill: nvidia-container-runtime did not terminate successfully: exit status 1: lstat /sys/fs/cgroup/devices/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod77f17d3d_8f49_4cd3_9768_1ade08f2724d.slice/docker-fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f.scope: no such file or directory\n: unknown"
sctr: unknown error after kill: nvidia-container-runtime did not terminate successfully: exit status 1: lstat /sys/fs/cgroup/devices/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod77f17d3d_8f49_4cd3_9768_1ade08f2724d.slice/docker-fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f.scope: no such file or directory
: unknown

해결​

4-4. Shim 프로세스 직접 삭제 (성공)

info

containerd-shim은 컨테이너와 1:1로 매핑되어 생명주기를 관리하는 독립적인 프로세스입니다.
특정 컨테이너의 Shim을 종료하더라도 해당 컨테이너의 관리 권한만 상실될 뿐, 호스트 내 다른 컨테이너나 Docker/containerd 데몬의 안정성에는 영향을 주지 않습니다.

  • PID 620710이 Shim 프로세스 PID를 확인하고 KILL
node19 ~ ➜ ps aux | grep $CONTAINER

root      620710  0.0  0.0 712448 10044 ?        Sl   Apr18   0:17 /usr/bin/containerd-shim-runc-v2 -namespace moby -id fdb38c96b7ecb9eff6d05fa0a9569813e41103ef008e3009f9ec0cb66cddbc1f -address /run/containerd/containerd.sock
root     2320661  0.0  0.0   8164  2376 pts/0    S+   04:50   0:00 grep --color=auto fdb38c96b7ec

node19 ~ ➜ kill -9 620710

다시 ctr을 통해 해당 컨테이너를 조회해보면 컨테이너가 삭제된 것을 확인할 수 있습니다.

node19 ~ ➜ ctr -n moby task ls | grep $CONTAINER

이제 다시 도커에서 컨테이너를 KILL하고 삭제합니다.
이전과 달리 별도 장애가 발생하지 않고 정상적으로 명령어가 수행되는 것을 확인할 수 있습니다.

node19 ~ ➜ docker kill $CONTAINER

fdb38c96b7ec

node19 ~ ➜ docker rm $CONTAINER

fdb38c96b7ec

마무리​

사실 리부팅하는게 제일 간편하지만, 실제로 운영 중인 상황이라면 컨테이너 하나에 문제가 생겼다고 무작정 재부팅할 수는 없습니다. 특히 저와같이 동일한 노드내에 다른 사용자들이 여전히 서비스를 이용 중이라면 더더욱 그렇습니다.

결국 리부팅 없이 문제를 해결하기 위해 시스템 내부 동작을 한 단계 더 깊이 이해하는 계기가 된것 같아 재밌었습니다.
