연습문제 1 : 스토리지 클래스​

다음 사양으로 local-sc라는 이름의 StorageClass를 생성하고 기본 스토리지 클래스로 설정합니다.

  • 제공자는 kubernetes.io/no-provisioner
  • 볼륨 바인딩 모드는 WaitForFirstConsumer이어야 합니다.
  • 볼륨 확장을 활성화해야 합니다.

연습문제 1 풀이​

Doc에서 Storage>Storage Classes페이지에서 예제를 가져옵니다.

YAML
local-sc.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: low-latency
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: csi-driver.example-vendor.example
reclaimPolicy: Retain # default value is Delete
allowVolumeExpansion: true
mountOptions:
  - discard # this might enable UNMAP / TRIM at the block storage layer
volumeBindingMode: WaitForFirstConsumer
parameters:
  guaranteedReadWriteLatency: "true" # provider-specific

해당 예제를 수정합니다.

  • 기본 스토리지 클래스 설정: storageclass.kubernetes.io/is-default-class를 true로 설정
  • provisioner를 kubernetes.io/no-provisioner로 변경
  • volumeBindingMode는 WaitForFirstConsumer
  • 볼륨 확장을 활성화는 allowVolumeExpansion 값이 true면 활성화
YAML
local-sc.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-sc
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: kubernetes.io/no-provisioner
reclaimPolicy: Retain
allowVolumeExpansion: true
mountOptions:
  - discard
volumeBindingMode: WaitForFirstConsumer
parameters:
  guaranteedReadWriteLatency: "true"
BASH
k apply -f local-sc.yaml
~ ➜ kubectl get storageclass
controlplane ~ ➜  kubectl get sc

NAME                    PROVISIONER                    RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
local-sc (default)   kubernetes.io/no-provisioner   Retain          WaitForFirstConsumer   true                   2s

연습문제 2: 사이드카 컨테이너​

네임스페이스 logging-ns에 logging-deployment이라는 이름의 deployment를 생성합니다.
이 deployment는 다음 사양으로 제공됩니다.

메인 컨테이너는 app-container로 명명하고 이미지 busybox를 사용하며 로그 작성을 시뮬레이션하기 위해 다음 명령을 실행해야합니다.

sh -c "while true; do echo 'Log entry' >> /var/log/app/app.log; sleep 5; done"

사이드카 컨테이너인 log-agent를 추가하고, 이 컨테이너는 bushbox이미지를 사용하고 다음 명령어를 실행해야합니다.

tail -f /var/log/app/app.log

log-agent로그는 메인 app-container에서 기록한 항목을 표시해야합니다.

  • 사이드카는 메인 컨테이너의 로그를 표시합니다.
  • 사이드카 컨테이너가 올바르게 구성되었나요?

연습문제 2 풀이​

문제는 하나의 Pod 내에서 두개의 컨테이너가 협업하는 사이드카 패턴을 구현하는 것이 핵심입니다.

  • app-container: 5초마다 로그를 생성하여 특정 경로(/var/log/app/app.log)에 기록합니다.
  • log-agent: 메인 컨테이너가 생성한 로그 파일을 실시간으로 트래킹(tail -f합니다. 서로 독립적인 파일 시스템을 가진 두 컨테이너가 동일한 파일을 공유해야 하므로, Pod 단위의 임시 볼륨인 emptyDir을 생성하고 두 컨테이너의 동일 경로에 마운트하여 문제를 해결할 수 있습니다.
  1. deployment 매니페스트 초안 생성
  • 디플로이먼트명 logging-deployment
  • 이미지 지정 --image busybox
  • 네임스페이스 지정 -n logging-ns
  • API 요청하지 않고 매니페스트만 가져오기 위한 옵션 추가 --dry-run=client
BASH
k create deployment logging-deployment --image busybox -n logging-ns --dry-run=client -o yaml > logging-deployment.yaml
~ ➜ kubectl get storageclass
YAML
logging-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: logging-deployment
  name: logging-deployment
  namespace: logging-ns
spec:
  replicas: 1
  selector:
    matchLabels:
      app: logging-deployment
  strategy: {}
  template:
    metadata:
      labels:
        app: logging-deployment
    spec:
      containers:
      - image: busybox
        name: busybox
        resources: {}
status: {}
BASH
k apply -f local-sc.yaml
  1. deployment 매니페스트 수정 생성한 매니페스트에 조건들을 추가합니다.
YAML
logging-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: logging-deployment
  name: logging-deployment
  namespace: logging-ns
spec:
  replicas: 1
  selector:
    matchLabels:
      app: logging-deployment
  strategy: {}
  template:
    metadata:
      labels:
        app: logging-deployment
    spec:
      containers:
      # 메인 컨테이너
      - image: busybox
        name: app-container
        command: ["sh", "-c", "while true; do echo 'Log entry' >> /var/log/app/app.log; sleep 5; done"]
        volumeMounts:
          - name: shared-volume
            mountPath: /var/log/app
        resources: {}
      # 사이드카 컨테이너 추가
      - image: busybox
        name: logging-agent
        command: ["sh", "-c", "tail -f /var/log/app/app.log"]
        volumeMounts:
          - name: shared-volume
            mountPath: /var/log/app
      volumes:
        - name: shared-volume
          emptyDir: {}
  1. deployment 생성
BASH
k apply -f logging-deployment.yaml

연습문제 5 : CSR​

Create a new user called john. Grant him access to the cluster using a csr named john-developer. Create a role developer which should grant John the permission to create, list, get, update and delete pods in the development namespace. The private key exists in the location: /root/CKA/john.key and csr at /root/CKA/john.csr.

Important Note: As of kubernetes 1.19, the CertificateSigningRequest object expects a signerName.
