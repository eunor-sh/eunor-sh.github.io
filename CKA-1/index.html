연습문제 1 : 멀티 컨테이너​

3개의 컨테이너가 있는 네임스페이스mc-namespace에 파드mc-pod를 생성합니다.
1번 컨테이너는 mc-pod-1로 명명하고 nginx:1-alpine이미지를 실행한 후 환경변수NODE_NAME를 노드 이름으로 설정합니다.
2번 컨테이너는 mc-pod-2로 명명하고, busybox:1이미지를 실행한 후, date명령어의 출력을 매초마다 /var/log/shared/date.log 파일에 지속적으로 기록해야 합니다.
3번 컨테이너는 mc-pod-3로 명명하고 busybox:1이미지를 실행하고, 2번째 컨테이너에서 생성된 date.log파일의 내용을 stdout으로 인쇄합니다. 단, 공유되는 비영구적 볼륨을 사용하세요.

  • POD 환경 변수가 올바르게 설정되었나요?
  • 사이드카 컨테이너에 로그가 표시되나요?

공유 볼륨으로 컨테이너 간 통신하기(mc-pod-2컨테이너의 date.log를 mc-pod-3에서 접근

  1. mc-namespace 확인
~ ➜ kubectl get ns mc-namespace
NAME           STATUS   AGE
mc-namespace   Active   17m
  1. mc-pod 매니페스트 작성

구조는 아래와 같이 mc-pod 내 mc-pod-1, mc-pod-2, mc-pod-3 총 1개의 Pod와 3개의 컨테이너를 생성해야한다.

mc-pod (pod)
├── mc-pod-1 (container)
├── mc-pod-2 (container)
└── mc-pod-3 (container)
YAML
mc-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: mc-pod
  namespace: mc-namespace
spec:
  containers:
    - name: mc-pod-1
      image: nginx:1-alpine 
      env:
        - name: NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName
    - name: mc-pod-2
      image: busybox:1
      volumeMounts:
        - name: shared-volume
          mountPath: /var/log/shared
      command:
        - "sh"
        - "-c"
        - "while true; do date >> /var/log/shared/date.log; sleep 1; done"
    - name: mc-pod-3
      image: busybox:1
      command:
        - "sh"
        - "-c"
        - "tail -f /var/log/shared/date.log"
      volumeMounts:
        - name: shared-volume
          mountPath: /var/log/shared
  volumes:
    - name: shared-volume
      emptyDir: {}
  1. mc-pod 생성
BASH
kubectl apply -f mc-pod.yaml

연습문제 4 : Service 생성​

port 6379의 클러스터 내 messaging pod를 노출시키기 위해 messaging-service 서비스를 만듭니다.
messaging pod는 default 네임스페이스에서 실행 중입니다.

  • 명령어를 사용합니다.
  • Service: messaging-service
  • Port: 6379
  • Type: ClusterIp
  • 올바른 라벨 사용

연습문제 4 풀이​

  1. messaging pod 확인 후 라벨 확인

pod 확인

~ ➜ kubectl get pod messaging
NAME        READY   STATUS    RESTARTS   AGE
messaging   1/1     Running   0          4m23s

옵션 --show-labels를 추가해 pod의 label 확인 tier=msg

~ ➜ kubectl get pod messaging --show-labels
controlplane ~ ➜  k get po messaging --show-labels 
NAME        READY   STATUS    RESTARTS   AGE     LABELS
messaging   1/1     Running   0          6m41s   tier=msg

  1. messaging-service 매니페스트 작성

    매니페스트 작성

BASH
messaging-service
apiVersion: v1
kind: Service
metadata:
  name: messaging-service
spec:
  selector:
    tier: msg
  ports:
    - protocol: TCP
      port: 6379
      targetPort: 6379
  type: ClusterIP
  1. 서비스(service) 생성 및 상태 조회 작성한 YAML 파일을 클러스터에 적용하여 messaging-service를 생성하고, 할당된 ClusterIP와 포트 정보를 확인합니다.
    1. 서비스 생성

    kubectl apply 명령어를 사용하여 정의된 매니페스트 파일을 실행합니다.

BASH
kubectl apply -f messaging-service.yaml
    1. 서비스 리스트 조회

    생성된 서비스가 정상적으로 클러스터 내부에 등록되었는지 확인합니다.

~ ➜ kubectl get service
controlplane ~ ➜  k get svc

kubernetes          ClusterIP   172.20.0.1       <none>        443/TCP    104m
messaging-service   ClusterIP   172.20.218.158   <none>        6379/TCP   3m28s
    1. 검토

    조회 결과에서 다음 항목들이 일치하는지 확인합니다.

    • TYPE : ClusterIP 클러스터 내부 통신 전용으로 설정되었습니다.
    • PORT(S) : 6379/TCP

  1. 서비스와 파드의 연결 상태 확인

    생성한 messaging-service가 실제 messaging 파드를 올바르게 타겟팅하고 있는지 확인합니다.
    서비스는 엔드포인트(EndPoints) 정보와 파드의 IP 주소를 대조하여 연결성을 검증합니다.

    1. 엔드포인트 확인

    먼저 kubectl get endpoints 명령어로 서비스에 바인딩된 IP를 확인합니다.

    엔드포인트에 표시된 172.17.0.9:6379가 실제 파드의 IP인지 확인하기 위해 -o wide 옵션을 사용하여 파드 상세 정보를 조회합니다.

~ ➜ kubectl get pod messaging -o wide
controlplane ~ ➜  k get po messaging -o wide

    • Service Endpoint : 172.17.0.9
    • Pod IP : 172.17.0.9

    두 IP가 동일한 것을 확인할 수 있습니다. 이를 통해 서비스의 셀렉터(Selector)가 파드를 정상적으로 감지하고 있으며, 서비스로 유입되는 트래픽이 messaging 파드의 6379 포트로 정확히 전달될 수 있는 상태임을 확인했습니다.

애플리케이션 orange가 배포되었지만, 문제가 있습니다.
문제를 식별하고 해결하세요.

  • 문제가 해결되었나요?

연습문제 6 풀이​

배포된 orange 파드가 정상적으로 가동되지 않는 문제를 해결해 봅니다. 이 과정에서 Init Container의 로그 확인 방법과 실행 중인 파드의 매니페스트를 수정하는 법을 익힐 수 있습니다.

1. 파드 상태 확인​

가장 먼저 파드의 현재 상태를 조회하여 어디서 문제가 발생했는지 확인합니다.

~ ➜ kubectl get pod orange
controlplane ~ ➜  k get po orange

NAME     READY   STATUS       RESTARTS       AGE
orange   0/1     Init:Error   5 (101s ago)   3m9s

STATUS가 Init:Error입니다. 이는 파드 기동 전 실행되는 초기화 컨테이너(Init Container)가 실패했음을 의미합니다.

2. 상세 로그 조사​

파드에는 여러 컨테이너가 포함될 수 있으므로, 로그를 볼 때 특정 컨테이너를 지정해야 합니다.

메인 컨테이너 로그 확인:

~ ➜ kubectl logs orange
controlplane ~ ➜  k logs orange 

Defaulted container "orange-container" out of: orange-container, init-myservice (init)
Error from server (BadRequest): container "orange-container" in pod "orange" is waiting to start: PodInitializing

메인 컨테이너인 orange-container는 시작도 못 하고 대기중(PodInitializing)인 것을 알 수 있습니다.

초기화 컨테이너(init-myservice) 로그 확인:

k logs orange init-myservice 

sh: sleeeep: not found

에러 로그(sh: sleeep: not fount)를 확인했으니, 해당 명령어를 올바른 명령어로 변경해줍니다.

변경을 위해 먼저 orange pod를 -o yaml 옵션을 통해 매니페스트를 추출합니다.

BASH
kubectl get pod orange -o yaml >> orange-pod.yaml

spec.initContainers.command 에서 sleeep 2;로 잘못된 명령어를 확인했고 sleep 2;로 변경합니다.

YAML
orange-pod.yaml
    ...

  initContainers:
  - command:
    - sh
    - -c
    - sleeeep 2; # sleep 2;로 교체
    image: busybox
    imagePullPolicy: Always
    name: init-myservice
    resources: {}
    terminationMessagePath: /dev/termination-log
    terminationMessagePolicy: File
    volumeMounts:
    - mountPath: /var/run/secrets/kubernetes.io/serviceaccount

  nodeName: controlplane

kubectl delete pod orange

연습문제 7: 서비스 노출​

이전 작업에서 생성된 hr-web-app을 클러스터 노드의 port 30082에서 액세스할 수 있는 hr-web-app-service라는 서비스로 노출하세요.

  • 웹 애플리케이션은 port 8080에서 대기합니다.

연습문제 7 풀이​

1. 대상 리소스의 라벨 확인​

서비스가 트래픽을 전달할 대상(Pod)을 정확히 찾기 위해 현재 hr-web-app의 라벨을 먼저 확인합니다. app=hr-web-app

~ ➜ kubectl get deployments --show-labels
controlplane ~ ➜  k get deployments.apps --show-labels 

NAME         READY   UP-TO-DATE   AVAILABLE   AGE     LABELS
hr-web-app   2/2     2            2           7m11s   app=hr-web-app

2. 서비스 생성 및 매니페스트 수정​

특정 nodePort를 지정하기 위해 create service 명령어를 사용한 뒤, 셀렉터를 수동으로 수정합니다.

  • 서비스 생성 명령어 실행(NodePort)
BASH
kubectl create service nodeport hr-web-app-service --tcp=8080:8080 --node-port=30082
  • YAML 추출 및 셀렉터 수정:
    명령어로 생성하면 셀렉터가 app: hr-web-app-service로 잡히기 때문에, 이를 실제 파드 라벨인 app: hr-web-app으로 변경해야 합니다.
    이후 라벨을 추가해주기 위해 hr-web-app-service의 매니페스트를 가져옵니다.
BASH
kubectl get service hr-web-app-service -o yaml >> hr-web-app-service.yaml
YAML
hr-web-app-service.yaml
apiVersion: v1
kind: Service
metadata:
  creationTimestamp: "2026-03-11T07:11:06Z"
  labels:
    app: hr-web-app-service
  name: hr-web-app-service
  namespace: default
  resourceVersion: "8209"
  uid: 90238eed-438e-423a-94e1-6407105129e8
spec:
  clusterIP: 172.20.84.2
  clusterIPs:
  - 172.20.84.2
  externalTrafficPolicy: Cluster
  internalTrafficPolicy: Cluster
  ipFamilies:
  - IPv4
  ipFamilyPolicy: SingleStack
  ports:
  - name: 8080-8080
    nodePort: 30082
    port: 8080
    protocol: TCP
    targetPort: 8080
  selector:
    app: hr-web-app-service  # app: hr-web-app으로 변경
  sessionAffinity: None
  type: NodePort
status:
  loadBalancer: {}
  • 설정 적용:
BASH
kubectl apply -f hr-web-app-service.yaml

  • 1. 서비스 엔드포인트 조회:
~ ➜ kubectl get endpoints hr-web-app-service
